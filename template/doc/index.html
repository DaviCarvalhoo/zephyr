This template ships a multi-tenant SaaS backend paired with a mobile app that's pre-wired for offline-first use, cloud sync, IAP, push, and health integrations. Everything below is shipped working — you can delete what you don't need.

Express + TypeScript + PostgreSQL + Knex. Multi-tenant accounts, role-based auth, S3 uploads, content publishing.

Web

Tooling

Post-install (Xcode + Android)

expo prebuild generates the iOS/Android projects, but a few capabilities have to be enabled by hand — Expo plugins can write entitlements but not check provisioning profiles.

iOS

  1. Open mobile/{{PROJECT_SLUG}}/ios/{{PROJECT_SLUG}}.xcworkspace in Xcode.
  2. Project → target {{PROJECT_SLUG}} → Signing & Capabilities.
  3. Pick your Team. Xcode will create a Bundle ID matching {{IOS_BUNDLE_ID}}.
  4. Click + Capability and add: Sign in with Apple, HealthKit, Push Notifications, and App Groups (if you use Live Activities).
  5. For each capability, also enable it in Apple Developer Portal under your App ID's capabilities — Xcode generates the entitlement file but won't sign with it unless the App ID has the matching capability.
  6. If you're shipping a Widget extension (Live Activity), add the same App Group to the widget target with the same group ID.

Android

  1. Open mobile/{{PROJECT_SLUG}}/android in Android Studio.
  2. Generate a release keystore: keytool -genkey -v -keystore release.jks -alias key -keyalg RSA -keysize 2048 -validity 10000.
  3. Create android/gradle.properties entries for {{PROJECT_SLUG_UPPER}}_UPLOAD_STORE_FILE, _KEY_ALIAS, _STORE_PASSWORD, _KEY_PASSWORD and reference them from android/app/build.gradle's signingConfigs.
  4. For Play Store submissions, drop your service account JSON at mobile/{{PROJECT_SLUG}}/google-play-key.json (or set GOOGLE_PLAY_SERVICE_ACCOUNT_JSON in server/.env and scripts/release-android.sh will extract it on each run).
  5. Health Connect users need Android 9+; users on older versions silently no-op. Don't drop minSdkVersion: 26 without a plan for the missing platform.

Heads up. The first expo prebuild generates fresh ios/ and android/ directories. scripts/install-native.sh patches them with our custom code; scripts/prebuild.sh refuses --clean so you don't accidentally wipe those edits.

Server architecture

See architecture.md at the repo root for the full breakdown. The short version:

  • Multi-tenant. users → user_in_accounts → accounts → domain entities. Every multi-tenant query filters by account_id.
  • Context-per-request DI. A fresh Context is built per request with all models instantiated; routes call req.context.user.foo().
  • Models own knex. Routes never hit knex directly — every DB op goes through a model method. Easier to test, easier to refactor.
  • Cookie auth for web, Bearer for mobile. Same JWT, different transport. try-set-user-by-token middleware checks both.
  • Refresh-token rotation. Mobile clients get a separate refresh token; /api/auth/refresh deletes the old RT and issues a new one (single-flighted on the server).

Auth flows

Email + password

POST /api/login    → { ok, user, token, refreshToken }
POST /api/signup   → { ok, user, account, token, refreshToken }
POST /api/auth/refresh
POST /api/logout

Apple Sign-In (iOS)

Client uses expo-apple-authentication, gets a native id_token, posts it to POST /api/auth/apple/mobile. Server verifies the token via Apple's JWKS (zero-dep), find- or-creates the user, returns the token pair.

Google (server-side flow)

WebBrowser.openAuthSessionAsync sends the user to /api/auth/google/app; server redirects to Google; Google redirects to /api/auth/google/callback; server exchanges code for id_token, verifies, mints our token pair, redirects to {{MOBILE_SCHEME}}://auth?token=…&refreshToken=…; client picks up the URL via Linking and calls continueWithToken.

Server-side keeps client_secret out of the app bundle and lets one Web OAuth client serve web + iOS + Android.

Offline DB + backup Phase 2

SQLite is the database; the server is a backup destination. UI screens never wait on the network for user data.

Two table classes

In src/services/db/migrations.ts:

  • USER_TABLES — tied to one account. Cleared on logout via clearUserData(); backed up to S3 for premium users.
  • SHARED_TABLES — re-syncable cache (e.g. content_catalog). Persists across logout — the next user skips re-downloading.

The write gate

setDbSigned(true) on login; setDbSigned(false) at the very start of logout, before table drops. User-scoped writes check demandSigned() first and silently no-op when the gate is closed — keeps in-flight renders from re-creating rows in tables we just dropped.

Backup flow

  1. POST /api/backup/upload-url → presigned PUT to staging key.
  2. PUT to S3 directly.
  3. POST /api/backup/promote → server copies staging → canonical.

The previous backup survives if the upload fails between steps. useAutoBackup in routes/index.tsx runs this on cold-start + every foreground transition, throttled to once per hour.

Content sync Phase 3

Server publishes a versioned catalog (server/content/catalog.json) — version is the SHA-256 of the file's bytes. Client probes /api/content/version on cold-start and every foreground transition; only pulls the catalog when the version actually changed.

Asset URLs

Large media (audio, images, videos) lives in S3. Client signs a request with HMAC-SHA256 over ${ts}:${contentPath} using a shared secret; server verifies, returns a presigned S3 URL cached for ~55 minutes. 

const url = await getAssetUrl('covers/welcome.jpg');
<Image source={{ uri: url }} />

IAP & premium Phase 4

react-native-iap v15 lazy-loaded with try/require so Expo Go gets graceful no-ops. 

const { activatePremium } = useAuth();
const purchase = await purchaseSubscription(PRODUCT_IDS.annual);
const result = await validateWithServer(purchase, token);
if (result.ok) await activatePremium(result.user);

refreshPremiumStatus() runs on every foreground transition (see <PremiumStatusRefresher />) — catches cancellations the user did externally (App Store / Play Store / website) without webhooks.

Server stub. core/iap-verify.ts returns active=true with a 30-day expiry and refuses to run in production. Implement Apple's App Store Server API and Google's Play Developer API before shipping anything that takes real money.

Notifications Phase 5

Custom-native modules — not expo-notifications. Two reasons:

  1. Cold-start tap dispatch. Native delegate fires before JS boots; we stash the tap in UserDefaults / SharedPreferences and dispatch a {{MOBILE_SCHEME}}://notif deep link.
  2. Per-button intents. Native APIs let action taps flip state without launching the app at all.

Files live in native/{ios,android}/; scripts/install-native.sh copies them into the prebuilt project dirs and prints the manual AppDelegate / MainActivity patches needed.

Health Phase 6

Unified surface over react-native-health (iOS HealthKit) and react-native-health-connect (Android Health Connect). Both libraries lazy-loaded; Expo Go gets a no-op service. 

await requestHealthPermissions();
const steps = await getStepsToday();
await writeExerciseSession({
    type: 'Yoga',
    startedAt: new Date(Date.now() - 1800_000),
    durationMs: 1800_000
});

iOS HealthKit needs an entitlement enabled in Apple Developer Portal under your App ID. Android Health Connect requires API 28+.

Gotchas (the ones we paid for)

  • NativeEventEmitter on iOS will crash if the native module doesn't extend RCTEventEmitter. Use Linking events instead.
  • App-extension CFBundleVersion must match the parent app's. scripts/bump-version.sh walks every Info.plist to keep them in sync.
  • expo prebuild --clean deletes hand-written native code. scripts/prebuild.sh refuses the flag.
  • NitroModules + Expo Go = crash on import. Lazy-load with try/require.
  • Single-flight refresh tokens: every successful refresh deletes the old RT. Concurrent callers must share one in-flight Promise or the second will see a deleted RT and trigger a false logout.

Code standards

See code.md at the repo root for the full list. The non-negotiables across server + mobile:

  • 4-space indent, semicolons, max 80 columns.
  • Always braces — no single-line if/for/while bodies.
  • Never silently catch — every catch at minimum logs.
  • Code in English; user-facing strings in Portuguese.
  • No any in TypeScript — unknown when truly unknown.
  • Routes go through models — never call knex directly from a route.
